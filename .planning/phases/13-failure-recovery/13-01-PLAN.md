<objective>
Add intelligent INVALID_FIELD failure recovery to run_soql_query. When a SOQL query fails with an invalid field error, the system automatically describes the failing object, fuzzy-matches the bad field name against actual fields using Levenshtein distance, and returns up to 3 ranked suggestions alongside the original error. The fresh describe result is stored in SchemaService cache automatically.

Purpose: AI agents currently get raw "No such column" errors with no guidance on correct field names. This phase turns failures into learning moments — the agent gets actionable suggestions AND the schema cache gets populated for future queries.

Output: Two new files (levenshtein.ts, levenshtein.test.ts), one new test file (failure-recovery.test.ts), one modified file (run_soql_query.ts). All 4 requirements (FAIL-01 through FAIL-04) fully covered with TDD.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Salesforce API → MCP Server | Error messages from Salesforce API cross into regex extraction. Error message content is untrusted. |
| MCP Server → AI Client | Field name suggestions are returned to the AI agent. These are actual Salesforce field names from describe results. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-01 | Information Disclosure | findSimilarFields suggestions | accept | Suggestions only contain actual field API names from connection.describe() — the same data an AI agent would get from salesforce_describe_object. No new information exposure. |
| T-13-02 | Tampering (ReDoS) | Regex `/No such column '(\w+)' on entity '(\w+)'/i` | mitigate | `\w+` is a non-backtracking quantifier with no nested alternation — not vulnerable to catastrophic backtracking. Regex operates on bounded SfError.message strings. |
| T-13-03 | Denial of Service | levenshtein() on large field lists | accept | Salesforce objects have finite fields (typically <500). Levenshtein per comparison is O(n*m) where n,m are field name lengths (~20 chars). Total cost: 500 * 20 * 20 = 200K operations — trivial, completes in <1ms. |
| T-13-04 | Spoofing | Crafted error message to trigger unexpected describe | accept | The describe call uses the object name extracted from the error message. Even if an attacker crafts the message, describe() only returns metadata for objects the connection has access to — bounded by Salesforce permissions. |
</threat_model>
